# Практическая работа 1

## Цель работы

Разработка базовой версии API для системы организации и проведения хакатонов с использованием фреймворка FastAPI. В рамках работы необходимо создать временную базу данных для основной сущности предметной области, реализовать вложенные модели и разработать API для работы с ними.

## Постановка задачи

По условию варианта требуется разработать программную систему для организации и проведения хакатонов.
В первой практической работе необходимо:

- создать временную базу данных для главной сущности;
- добавить 2–3 тестовые записи;
- реализовать один вложенный объект;
- реализовать список вложенных объектов;
- описать модели данных;
- реализовать API для вложенного объекта.

## Описание предметной области

В качестве предметной области я выбрала систему управления хакатонами. Основной сущностью проекта стал **Hackathon**.

Для выполнения требований задания я выделила следующие связанные сущности:

- **Organizer** — организатор хакатона, вложенный одиночный объект;
- **ChallengeTask** — задача хакатона, список вложенных объектов.

Таким образом, сущность `Hackathon` содержит:

- основную информацию о мероприятии;
- одного организатора;
- список задач, публикуемых для участников.

## Используемые технологии

В работе используются следующие технологии:

- **FastAPI** — фреймворк для создания API;
- **Pydantic** — библиотеку для описания моделей данных и валидации;
- **Uvicorn** — ASGI-сервер для запуска приложения.

## Разработка моделей данных

В рамках первой практической работы я разработала три основные модели.

### 1. Organizer

Модель `Organizer` описывает организатора хакатона и содержит следующие поля:

- `id` — идентификатор;
- `full_name` — ФИО организатора;

- `phone` — контактный телефон;
- `organization` — организация, представляющая организатора.

### 2. ChallengeTask

Модель `ChallengeTask` описывает задачу, опубликованную в рамках хакатона. Она включает:

### 3. Hackathon

Модель `Hackathon` является главной сущностью приложения и содержит:

## Реализованный API

В приложении я реализовала эндпоинты для работы с главной сущностью `Hackathon`, а также API для вложенного объекта `Organizer`.

### Эндпоинты для Hackathon

- `GET /hackathons` — получить список хакатонов;
- `GET /hackathons/{hackathon_id}` — получить информацию о конкретном хакатоне по идентификатору;
- `POST /hackathons` — создать новый хакатон;
- `PUT /hackathons/{hackathon_id}` — обновить существующий хакатон;
- `DELETE /hackathons/{hackathon_id}` — удалить хакатон.

### Эндпоинты для Organizer

- `GET /organizers` — получить список организаторов;
- `GET /organizers/{organizer_id}` — получить данные организатора по идентификатору;
- `POST /organizers` — создать нового организатора;
- `PUT /organizers/{organizer_id}` — обновить информацию об организаторе.

# Практическая работа 3

## Цель работы

Настройка системы миграций базы данных с использованием Alembic, а также перенос конфиденциальных параметров подключения к PostgreSQL в `.env` файл. Дополнительно мне нужно было реализовать изменение структуры базы данных через миграции и проверить применение изменений на практике.

## Постановка задачи

В третьем практическом задании требовалось:

- подключить и настроить Alembic в проекте;
- связать Alembic с моделями SQLModel;
- настроить получение URL базы данных из `.env`;
- исключить `.env` из индексации через `.gitignore`;
- внести изменение в модель связи many-to-many;
- сгенерировать и применить миграцию;
- проверить результат в PostgreSQL через pgAdmin.

## Настройка Alembic

Для реализации миграций в проект я добавила библиотеку **Alembic**. После её установки я выполнила инициализацию структуры миграций командой:

```bash
alembic init migrations
```

В результате в проекте появились:

- файл `alembic.ini`;
- папка `migrations`;
- файл `migrations/env.py`;
- папка `migrations/versions`;
- шаблон `script.py.mako`.

## Настройка окружения и подключение к базе данных

Для безопасного хранения строки подключения к базе данных я создала `.env` файл. В нём была размещена переменная окружения с URL подключения к PostgreSQL.

В файле `connection.py` я изменила подключение к базе данных таким образом, чтобы строка подключения считывалась не из кода напрямую, а из переменной окружения. Для этого я использовала библиотеку `python-dotenv`.

Также я создала файл `.gitignore`, в который добавила строку:

```text
*.env
```

Это позволило исключить `.env` из индексации Git и избежать попадания конфиденциальных данных в репозиторий.

## Настройка Alembic для работы с SQLModel

В файле `migrations/env.py` я выполнила следующие изменения:

- импортировала модели из `models.py`;
- передала в переменную `target_metadata` значение `SQLModel.metadata`;
- настроила чтение строки подключения к базе данных из `.env`.

Эти изменения были сделаны для того, чтобы Alembic мог сравнивать текущую структуру базы данных с моделями приложения и автоматически генерировать файлы миграций.

Также в файл `script.py.mako` я добавила импорт библиотеки `sqlmodel`, обеспечив корректную генерацию миграций для моделей SQLModel.

## Изменение модели связи many-to-many

Была доработана промежуточная модель `ParticipantHackathonLink`, реализующую связь many-to-many между участниками и хакатонами.

Ранее она содержала только два внешних ключа:

- `participant_id`
- `hackathon_id`

В третьей практике я добавила в модель дополнительное поле:

- `role_in_hackathon`

Это поле позволяет хранить роль участника в конкретном хакатоне, например: разработчик, дизайнер, аналитик или капитан команды.

Так ассоциативная сущность стала содержать не только технические ключи связи, но и дополнительную предметную информацию.

## Создание и применение миграции

После изменения модели я создала миграцию с помощью команды:

```bash
alembic revision --autogenerate -m "add role_in_hackathon to participant hackathon link"
```

В результате в папке `migrations/versions` был создан новый файл, содержащий описание изменений схемы базы данных.

Затем я применила миграцию к базе данных командой:

```bash
alembic upgrade head
```

## Реализация регистрации, авторизации и JWT-аутентификации

## Что нужно было реализовать

Чтобы довести проект до уровня 15 баллов, нужно было добавить:

## Добавление модели пользователя

Для начала я добавила в проект отдельную модель `User`, которая отвечает именно за пользовательские аккаунты.

В ней я предусмотрела следующие поля:

- `id` — идентификатор пользователя;

После этого в базе данных появилась отдельная таблица пользователей, с которой уже можно было строить регистрацию и авторизацию.

## Как я организовала хранение паролей

Одним из самых важных моментов было то, что пароль нельзя хранить в базе в обычном виде. Поэтому было реализовано хэширование пароля.

Это означает, что при регистрации в таблицу `User` сохраняется не сам пароль, а его безопасное преобразованное значение — хэш. Когда пользователь потом пытается войти в систему, введённый пароль не сравнивается напрямую с текстом из базы, а проверяется через функцию сравнения с уже сохранённым хэшем.

Такой подход уже соответствует нормальной практике разработки серверных приложений.


## Реализация JWT-аутентификации

После этого я добавила JWT-аутентификацию:

1. пользователь регистрируется в системе;
2. затем выполняет вход по имени пользователя и паролю;
3. если данные введены правильно, сервер создаёт JWT-токен;
4. этот токен используется для доступа к защищённым эндпоинтам;
5. при обращении к таким маршрутам приложение проверяет токен и определяет, кто именно отправил запрос.

Таким образом, мне удалось реализовать не просто вход в систему, а схему авторизации через токены доступа.



## Выделение логики безопасности в отдельный файл

В этом файле я собрала:

- хэширование пароля;
- проверку пароля;
- создание JWT-токена;
- декодирование и проверку токена;
- получение текущего пользователя;
- проверку активности пользователя.



## Добавленные эндпоинты

### Регистрация пользователя

- `POST /auth/register`

Эндпоинт используется для создания нового пользователя. Перед сохранением пользователя в базе проверяется, что имя и электронная почта ещё не заняты. После этого пароль хэшируется, и новая запись добавляется в таблицу `User`.

### Авторизация пользователя

- `POST /auth/login`

Эндпоинт нужен для входа в систему. Пользователь передаёт имя и пароль, приложение проверяет их корректность и в случае успеха возвращает JWT-токен.

### Получение информации о текущем пользователе

- `GET /users/me`

Метод позволяет получить данные о пользователе, который сейчас авторизован. Он полезен для проверки того, что токен действительно работает и пользователь успешно распознаётся системой.

### Получение списка пользователей

- `GET /users`

Эндпоинт возвращает список зарегистрированных пользователей. Доступ к нему возможен только после успешной аутентификации.

### Смена пароля

- `POST /users/change-password`

Метод позволяет пользователю изменить свой пароль. Для этого он должен указать старый пароль и новый. Сначала приложение проверяет старый пароль, а затем сохраняет новый уже в виде хэша.



## Как это работает на практике

В итоге сценарий работы с пользователем выглядит так:

1. отправляю запрос на регистрацию;

3. выполняю вход;
4. получаю JWT-токен;
5. с этим токеном могу обращаться к защищённым эндпоинтам;
6. могу посмотреть данные своей учётной записи или изменить пароль.
