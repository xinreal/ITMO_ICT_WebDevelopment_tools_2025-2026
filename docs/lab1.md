## Практическая работа 1

### Цель работы

Целью данной лабораторной работы является разработка базовой версии API для системы организации и проведения хакатонов с использованием фреймворка FastAPI. В рамках работы необходимо создать временную базу данных для основной сущности предметной области, реализовать вложенные модели и разработать API для работы с ними.


### Постановка задачи

По условию варианта требуется разработать программную систему для организации и проведения хакатонов. Хакатон представляет собой мероприятие, в рамках которого участники объединяются в команды и в течение ограниченного времени работают над решением поставленных задач или созданием новых проектов.

В первой работе необходимо:

- создать временную базу данных для главной сущности;
- добавить 2–3 тестовые записи;
- реализовать один вложенный объект;
- реализовать список вложенных объектов;
- описать модели данных;
- реализовать API для вложенного объекта.


### Описание предметной области

В качестве предметной области была выбрана система управления хакатонами.  
Основной сущностью проекта является **Hackathon**.

Для выполнения требований задания были выделены следующие связанные сущности:

- **Organizer** — организатор хакатона, вложенный одиночный объект;
- **ChallengeTask** — задача хакатона, список вложенных объектов.

Таким образом, сущность `Hackathon` содержит:
- основную информацию о мероприятии;
- одного организатора;
- список задач, публикуемых для участников.


### Используемые технологии

В работе использовались следующие технологии:

- **Python** — основной язык разработки;
- **FastAPI** — фреймворк для создания API;
- **Pydantic** — библиотека для описания моделей данных и валидации;
- **Uvicorn** — ASGI-сервер для запуска приложения.


### Структура проекта

Проект имеет следующую структуру:

```text
hackathon_api/

│   ├── main.py

```

**Назначение файлов**

- **main.py** — содержит основной FastAPI-приложение и API-эндпоинты;
- **models.py** — содержит модели данных, описанные с помощью Pydantic;
- **temp_db.py** — содержит временную базу данных в виде списков объектов.


### Разработка моделей данных

В рамках лабораторной работы были разработаны три основные модели.

**1. Organizer**

Модель `Organizer` описывает организатора хакатона и содержит следующие поля:

- `id` — идентификатор;
- `full_name` — ФИО организатора;
- `email` — адрес электронной почты;
- `phone` — контактный телефон;
- `organization` — организация, представляющая организатора.

**2. ChallengeTask**

Модель `ChallengeTask` описывает задачу, опубликованную в рамках хакатона. Она включает:

- `id` — идентификатор задачи;
- `title` — название задачи;
- `description` — описание задачи;
- `requirements` — требования к решению;
- `evaluation_criteria` — критерии оценки.

**3. Hackathon**

Модель `Hackathon` является главной сущностью приложения и содержит:

- `id` — идентификатор хакатона;
- `title` — название;
- `description` — описание;
- `city` — город проведения;
- `format` — формат проведения;
- `duration_hours` — продолжительность в часах;
- `organizer` — вложенный объект типа `Organizer`;
- `tasks` — список объектов типа `ChallengeTask`.

Таким образом, требования задания были выполнены: у главной сущности присутствует один вложенный объект и список вложенных объектов.


### Временная база данных

На первом этапе работы вместо реальной базы данных использовалось временное хранилище данных. Оно было реализовано в файле `temp_db.py` в виде списков Python-объектов.

Во временной базе были созданы тестовые записи для:

- организаторов;
- хакатонов.

Каждый хакатон содержит вложенный объект организатора и список задач. Это позволило реализовать структуру данных, близкую к реальной предметной области, без подключения полноценной СУБД.


### Реализованный API

В приложении были реализованы эндпоинты для работы с главной сущностью `Hackathon`, а также API для вложенного объекта `Organizer`.

**Эндпоинты для Hackathon**

- `GET /hackathons` — получить список хакатонов;
- `GET /hackathons/{hackathon_id}` — получить информацию о конкретном хакатоне по идентификатору;
- `POST /hackathons` — создать новый хакатон;
- `PUT /hackathons/{hackathon_id}` — обновить существующий хакатон;
- `DELETE /hackathons/{hackathon_id}` — удалить хакатон.

**Эндпоинты для Organizer**

- `GET /organizers` — получить список организаторов;
- `GET /organizers/{organizer_id}` — получить данные организатора по идентификатору;
- `POST /organizers` — создать нового организатора;
- `PUT /organizers/{organizer_id}` — обновить информацию об организаторе.

## Практическая работа 2

### Цель работы

Целью второй практической работы являлось расширение первой части проекта за счёт подключения реальной базы данных PostgreSQL, разработки моделей SQLModel и реализации API для работы с сущностями предметной области. Дополнительно необходимо было настроить связь many-to-many и реализовать вложенное отображение связанных данных.


### Постановка задачи

Во втором практическом задании требовалось:

- пошагово реализовать подключение к базе данных;
- создать модели на основе своего варианта;
- разработать API для работы с моделями;
- реализовать связь many-to-many;
- обеспечить вложенное отображение связанных сущностей в ответах API.

В качестве предметной области по-прежнему использовалась система организации и проведения хакатонов.


### Подключение базы данных

Для хранения данных была использована система управления базами данных PostgreSQL.  
Сначала в pgAdmin была создана база данных `hackathon_db`, после чего приложение было подключено к ней через SQLModel.

Подключение было реализовано в файле `connection.py`, где были описаны:

- строка подключения к базе данных;
- объект `engine`;
- функция для создания таблиц;
- функция для получения сессии.

После запуска приложения таблицы автоматически создавались в PostgreSQL на основе описанных моделей.


### Структура проекта

Во второй части работы структура проекта была изменена следующим образом:

```text
practice_2/
├── __init__.py
├── main.py
├── connection.py
└── models.py
```

**Назначение файлов**

- **models.py** — описание моделей базы данных;
- **connection.py** — подключение к PostgreSQL, создание `engine`, инициализация таблиц и работа с сессией;
- **main.py** — основное FastAPI-приложение, схемы запросов и ответов, API-эндпоинты;
- **__init__.py** — служебный файл для корректной работы структуры проекта как пакета.


### Разработка моделей

Во второй части были реализованы следующие модели.

**1. Organizer**

Модель `Organizer` описывает организатора хакатона и содержит:

- `id`
- `full_name`
- `email`
- `phone`
- `organization`

Один организатор может быть связан с несколькими хакатонами.

**2. Participant**

Модель `Participant` описывает участника хакатона и содержит:

- `id`
- `full_name`
- `email`
- `contact_number`
- `specialization`

Один участник может принимать участие в нескольких хакатонах.

**3. Hackathon**

Модель `Hackathon` является центральной сущностью и содержит:

- `id`
- `title`
- `description`
- `city`
- `format`
- `duration_hours`
- `organizer_id`

Каждый хакатон связан с одним организатором, а также может содержать множество участников и задач.

**4. ChallengeTask**

Модель `ChallengeTask` описывает задачу хакатона и включает:

- `id`
- `title`
- `description`
- `requirements`
- `evaluation_criteria`
- `hackathon_id`

Каждая задача относится к одному хакатону.

**5. ParticipantHackathonLink**

Модель `ParticipantHackathonLink` представляет собой промежуточную таблицу для реализации связи many-to-many между участниками и хакатонами.

Она содержит два поля:

- `participant_id`
- `hackathon_id`


### Реализация связи many-to-many

Связь many-to-many была реализована между сущностями:

- **Participant**
- **Hackathon**

Это означает, что:

- один участник может участвовать в нескольких хакатонах;
- один хакатон может включать нескольких участников.

Для реализации данной связи была создана отдельная промежуточная таблица `ParticipantHackathonLink`.


### Разработка API

В приложении были реализованы API-эндпоинты для основных сущностей.

**API для Organizer**

- `POST /organizers/` — создание организатора;
- `GET /organizers/` — получение списка организаторов;
- `GET /organizers/{organizer_id}` — получение организатора по идентификатору;
- `PUT /organizers/{organizer_id}` — обновление организатора;
- `DELETE /organizers/{organizer_id}` — удаление организатора.

**API для Participant**

- `POST /participants/` — создание участника;
- `GET /participants/` — получение списка участников;
- `GET /participants/{participant_id}` — получение участника по идентификатору;
- `PUT /participants/{participant_id}` — обновление участника;
- `DELETE /participants/{participant_id}` — удаление участника.

**API для Hackathon**

- `POST /hackathons/` — создание хакатона;
- `GET /hackathons/` — получение списка хакатонов;
- `GET /hackathons/{hackathon_id}` — получение хакатона по идентификатору;
- `PUT /hackathons/{hackathon_id}` — обновление хакатона;
- `DELETE /hackathons/{hackathon_id}` — удаление хакатона.

**API для ChallengeTask**

- `POST /tasks/` — создание задачи;
- `GET /tasks/` — получение списка задач;
- `GET /tasks/{task_id}` — получение задачи по идентификатору;
- `PUT /tasks/{task_id}` — обновление задачи;
- `DELETE /tasks/{task_id}` — удаление задачи.


### Реализация many-to-many через API

Для добавления участника в хакатон был реализован отдельный эндпоинт:

- `POST /hackathons/{hackathon_id}/participants/{participant_id}`

Данный метод добавляет запись в промежуточную таблицу `ParticipantHackathonLink`, тем самым связывая участника с конкретным хакатоном.


### Вложенное отображение

В рамках задания требовалось реализовать вложенное отображение связанных сущностей.

Под вложенным отображением понимается такой формат ответа API, при котором в объекте хакатона возвращаются не только идентификаторы связанных сущностей, но и сами связанные объекты.

Для этого был реализован эндпоинт:

- `GET /hackathons/{hackathon_id}/full`

Он возвращает:

- информацию о хакатоне;
- вложенный объект организатора;
- список участников;
- список задач.

Таким образом, данные возвращаются в удобном и наглядном виде, что соответствует требованиям задания.
