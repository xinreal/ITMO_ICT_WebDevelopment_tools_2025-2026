# Практическая работа 1

## Цель работы

Целью моей работы являлась разработка базовой версии API для системы организации и проведения хакатонов с использованием фреймворка FastAPI. В рамках работы мне необходимо было создать временную базу данных для основной сущности предметной области, реализовать вложенные модели и разработать API для работы с ними.

## Постановка задачи

По условию варианта мне требовалось разработать программную систему для организации и проведения хакатонов. Хакатон представляет собой мероприятие, в рамках которого участники объединяются в команды и в течение ограниченного времени работают над решением поставленных задач или созданием новых проектов.

В первой практической работе мне необходимо было:

- создать временную базу данных для главной сущности;
- добавить 2–3 тестовые записи;
- реализовать один вложенный объект;
- реализовать список вложенных объектов;
- описать модели данных;
- реализовать API для вложенного объекта.

## Описание предметной области

В качестве предметной области я выбрала систему управления хакатонами. Основной сущностью проекта стал **Hackathon**.

Для выполнения требований задания я выделила следующие связанные сущности:

- **Organizer** — организатор хакатона, вложенный одиночный объект;
- **ChallengeTask** — задача хакатона, список вложенных объектов.

Таким образом, сущность `Hackathon` содержит:

- основную информацию о мероприятии;
- одного организатора;
- список задач, публикуемых для участников.

## Используемые технологии

В работе я использовала следующие технологии:

- **Python** — основной язык разработки;
- **FastAPI** — фреймворк для создания API;
- **Pydantic** — библиотеку для описания моделей данных и валидации;
- **Uvicorn** — ASGI-сервер для запуска приложения.

## Структура проекта

Проект в первой части работы имеет следующую структуру:

```text
hackathon_api/
├── app/
│   ├── __init__.py
│   ├── main.py
│   ├── models.py
│   └── temp_db.py
```

### Назначение файлов

- **main.py** — содержит основное FastAPI-приложение и API-эндпоинты;
- **models.py** — содержит модели данных, описанные с помощью Pydantic;
- **temp_db.py** — содержит временную базу данных в виде списков объектов.

## Разработка моделей данных

В рамках первой практической работы я разработала три основные модели.

### 1. Organizer

Модель `Organizer` описывает организатора хакатона и содержит следующие поля:

- `id` — идентификатор;
- `full_name` — ФИО организатора;
- `email` — адрес электронной почты;
- `phone` — контактный телефон;
- `organization` — организация, представляющая организатора.

### 2. ChallengeTask

Модель `ChallengeTask` описывает задачу, опубликованную в рамках хакатона. Она включает:

- `id` — идентификатор задачи;
- `title` — название задачи;
- `description` — описание задачи;
- `requirements` — требования к решению;
- `evaluation_criteria` — критерии оценки.

### 3. Hackathon

Модель `Hackathon` является главной сущностью приложения и содержит:

- `id` — идентификатор хакатона;
- `title` — название;
- `description` — описание;
- `city` — город проведения;
- `format` — формат проведения;
- `duration_hours` — продолжительность в часах;
- `organizer` — вложенный объект типа `Organizer`;
- `tasks` — список объектов типа `ChallengeTask`.

Таким образом, я выполнила требования задания: у главной сущности присутствует один вложенный объект и список вложенных объектов.

## Временная база данных

На первом этапе работы вместо реальной базы данных я использовала временное хранилище данных. Оно было реализовано в файле `temp_db.py` в виде списков Python-объектов.

Во временной базе данных я создала тестовые записи для:

- организаторов;
- хакатонов.

Каждый хакатон содержит вложенный объект организатора и список задач. Это позволило мне реализовать структуру данных, близкую к реальной предметной области, без подключения полноценной СУБД.

## Реализованный API

В приложении я реализовала эндпоинты для работы с главной сущностью `Hackathon`, а также API для вложенного объекта `Organizer`.

### Эндпоинты для Hackathon

- `GET /hackathons` — получить список хакатонов;
- `GET /hackathons/{hackathon_id}` — получить информацию о конкретном хакатоне по идентификатору;
- `POST /hackathons` — создать новый хакатон;
- `PUT /hackathons/{hackathon_id}` — обновить существующий хакатон;
- `DELETE /hackathons/{hackathon_id}` — удалить хакатон.

### Эндпоинты для Organizer

- `GET /organizers` — получить список организаторов;
- `GET /organizers/{organizer_id}` — получить данные организатора по идентификатору;
- `POST /organizers` — создать нового организатора;
- `PUT /organizers/{organizer_id}` — обновить информацию об организаторе.

## Тестирование приложения

Для проверки работы API я использовала автоматически сгенерированную документацию Swagger UI, доступную по адресу:

## Результаты работы

В ходе выполнения первой практической работы я разработала базовый API для хакатонов. Я реализовала временную базу данных, содержащую тестовые записи, создала вложенные модели и настроила основные эндпоинты.

В качестве главной сущности я выбрала хакатон, что позволило естественным образом представить требуемую структуру данных: один вложенный объект (`Organizer`) и список объектов (`ChallengeTask`).

# Практическая работа 2

## Цель работы

Целью второй практической работы являлось расширение первой части проекта за счёт подключения реальной базы данных PostgreSQL, разработки моделей SQLModel и реализации API для работы с сущностями предметной области. Дополнительно мне необходимо было настроить связь many-to-many и реализовать вложенное отображение связанных данных.

## Постановка задачи

Во втором практическом задании мне требовалось:

- пошагово реализовать подключение к базе данных;
- создать модели на основе своего варианта;
- разработать API для работы с моделями;
- реализовать связь many-to-many;
- обеспечить вложенное отображение связанных сущностей в ответах API.

В качестве предметной области по-прежнему использовалась система организации и проведения хакатонов.

## Подключение базы данных

Для хранения данных я использовала систему управления базами данных PostgreSQL. Сначала в pgAdmin я создала базу данных `hackathon_db`, после чего подключила к ней приложение через SQLModel.

Подключение было реализовано в файле `connection.py`, где я описала:

- строку подключения к базе данных;
- объект `engine`;
- функцию для создания таблиц;
- функцию для получения сессии.

После запуска приложения таблицы автоматически создавались в PostgreSQL на основе описанных моделей.

## Структура проекта

Во второй части работы структура проекта была изменена следующим образом:

```text
practice_2/
├── __init__.py
├── main.py
├── connection.py
└── models.py
```

### Назначение файлов

- **models.py** — описание моделей базы данных;
- **connection.py** — подключение к PostgreSQL, создание `engine`, инициализация таблиц и работа с сессией;
- **main.py** — основное FastAPI-приложение, схемы запросов и ответов, API-эндпоинты;
- **__init__.py** — служебный файл для корректной работы структуры проекта как пакета.

## Разработка моделей

Во второй части работы я реализовала следующие модели.

### 1. Organizer

Модель `Organizer` описывает организатора хакатона и содержит:

- `id`
- `full_name`
- `email`
- `phone`
- `organization`

Один организатор может быть связан с несколькими хакатонами.

### 2. Participant

Модель `Participant` описывает участника хакатона и содержит:

- `id`
- `full_name`
- `email`
- `contact_number`
- `specialization`

Один участник может принимать участие в нескольких хакатонах.

### 3. Hackathon

Модель `Hackathon` является центральной сущностью и содержит:

- `id`
- `title`
- `description`
- `city`
- `format`
- `duration_hours`
- `organizer_id`

Каждый хакатон связан с одним организатором, а также может содержать множество участников и задач.

### 4. ChallengeTask

Модель `ChallengeTask` описывает задачу хакатона и включает:

- `id`
- `title`
- `description`
- `requirements`
- `evaluation_criteria`
- `hackathon_id`

Каждая задача относится к одному хакатону.

### 5. ParticipantHackathonLink

Модель `ParticipantHackathonLink` представляет собой промежуточную таблицу для реализации связи many-to-many между участниками и хакатонами.

Она содержит два поля:

- `participant_id`
- `hackathon_id`

## Реализация связи many-to-many

Связь many-to-many я реализовала между сущностями:

- **Participant**
- **Hackathon**

Это означает, что:

- один участник может участвовать в нескольких хакатонах;
- один хакатон может включать нескольких участников.

Для реализации данной связи я создала отдельную промежуточную таблицу `ParticipantHackathonLink`.

## Разработка API

Во второй части работы я реализовала API-эндпоинты для основных сущностей.

### API для Organizer

- `POST /organizers/` — создание организатора;
- `GET /organizers/` — получение списка организаторов;
- `GET /organizers/{organizer_id}` — получение организатора по идентификатору;
- `PUT /organizers/{organizer_id}` — обновление организатора;
- `DELETE /organizers/{organizer_id}` — удаление организатора.

### API для Participant

- `POST /participants/` — создание участника;
- `GET /participants/` — получение списка участников;
- `GET /participants/{participant_id}` — получение участника по идентификатору;
- `PUT /participants/{participant_id}` — обновление участника;
- `DELETE /participants/{participant_id}` — удаление участника.

### API для Hackathon

- `POST /hackathons/` — создание хакатона;
- `GET /hackathons/` — получение списка хакатонов;
- `GET /hackathons/{hackathon_id}` — получение хакатона по идентификатору;
- `PUT /hackathons/{hackathon_id}` — обновление хакатона;
- `DELETE /hackathons/{hackathon_id}` — удаление хакатона.

### API для ChallengeTask

- `POST /tasks/` — создание задачи;
- `GET /tasks/` — получение списка задач;
- `GET /tasks/{task_id}` — получение задачи по идентификатору;
- `PUT /tasks/{task_id}` — обновление задачи;
- `DELETE /tasks/{task_id}` — удаление задачи.

## Реализация many-to-many через API

Для добавления участника в хакатон я реализовала отдельный эндпоинт:

- `POST /hackathons/{hackathon_id}/participants/{participant_id}`

Данный метод добавляет запись в промежуточную таблицу `ParticipantHackathonLink`, тем самым связывая участника с конкретным хакатоном.

## Вложенное отображение

Для этого я реализовала эндпоинт:

Он возвращает:

# Практическая работа 3

## Цель работы

Целью третьей практической работы являлась настройка системы миграций базы данных с использованием Alembic, а также перенос конфиденциальных параметров подключения к PostgreSQL в `.env` файл. Дополнительно мне необходимо было реализовать изменение структуры базы данных через миграции и проверить применение изменений на практике.

## Постановка задачи

В третьем практическом задании мне требовалось:

- подключить и настроить Alembic в проекте;
- связать Alembic с моделями SQLModel;
- настроить получение URL базы данных из `.env`;
- исключить `.env` из индексации через `.gitignore`;
- внести изменение в модель связи many-to-many;
- сгенерировать и применить миграцию;
- проверить результат в PostgreSQL через pgAdmin.

## Настройка Alembic

Для реализации миграций в проект я добавила библиотеку **Alembic**. После её установки я выполнила инициализацию структуры миграций командой:

```bash
alembic init migrations
```

В результате в проекте появились:

- файл `alembic.ini`;
- папка `migrations`;
- файл `migrations/env.py`;
- папка `migrations/versions`;
- шаблон `script.py.mako`.

## Настройка окружения и подключение к базе данных

Для безопасного хранения строки подключения к базе данных я создала `.env` файл. В нём была размещена переменная окружения с URL подключения к PostgreSQL.

В файле `connection.py` я изменила подключение к базе данных таким образом, чтобы строка подключения считывалась не из кода напрямую, а из переменной окружения. Для этого я использовала библиотеку `python-dotenv`.

Также я создала файл `.gitignore`, в который добавила строку:

```text
*.env
```

Это позволило исключить `.env` из индексации Git и избежать попадания конфиденциальных данных в репозиторий.

## Настройка Alembic для работы с SQLModel

В файле `migrations/env.py` я выполнила следующие изменения:

- импортировала модели из `models.py`;
- передала в переменную `target_metadata` значение `SQLModel.metadata`;
- настроила чтение строки подключения к базе данных из `.env`.

Эти изменения были необходимы для того, чтобы Alembic мог сравнивать текущую структуру базы данных с моделями приложения и автоматически генерировать файлы миграций.

Дополнительно в файл `script.py.mako` я добавила импорт библиотеки `sqlmodel`, что обеспечивает корректную генерацию миграций для моделей SQLModel.

## Изменение модели связи many-to-many

В рамках практической работы я доработала промежуточную модель `ParticipantHackathonLink`, реализующую связь many-to-many между участниками и хакатонами.

Ранее она содержала только два внешних ключа:

- `participant_id`
- `hackathon_id`

В третьей практике я добавила в модель дополнительное поле:

- `role_in_hackathon`

Это поле позволяет хранить роль участника в конкретном хакатоне, например: разработчик, дизайнер, аналитик или капитан команды.

Таким образом, ассоциативная сущность стала содержать не только технические ключи связи, но и дополнительную предметную информацию.

## Создание и применение миграции

После изменения модели я создала миграцию с помощью команды:

```bash
alembic revision --autogenerate -m "add role_in_hackathon to participant hackathon link"
```

В результате в папке `migrations/versions` был создан новый файл миграции, содержащий описание изменений схемы базы данных.

Затем я применила миграцию к базе данных командой:

```bash
alembic upgrade head
```

После этого в таблице `participanthackathonlink` в PostgreSQL появился новый столбец `role_in_hackathon`.

## Проверка результата

Проверку выполнения миграции я проводила через pgAdmin. После применения команды `alembic upgrade head` в структуре таблицы `participanthackathonlink` появился новый столбец `role_in_hackathon`.

Это подтвердило, что:

- Alembic был корректно настроен;
- миграции успешно создаются;
- изменения моделей могут быть перенесены в базу данных без ручного редактирования таблиц.

## Результаты работы

В ходе выполнения третьей практической работы я добавила в проект поддержку миграций базы данных. Я настроила Alembic, `.env` и `.gitignore`, а также реализовала передачу строки подключения к PostgreSQL через переменные окружения.

Кроме того, я изменила промежуточную таблицу связи many-to-many, после чего с помощью Alembic создала и применила миграцию, обновившую структуру базы данных.
